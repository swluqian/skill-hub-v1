# 技能更新版本可见性方案

## 问题描述

当用户更新已发布的 Skill 时，当前实现会直接修改原记录的状态为 `pending`，导致：

- Skill 广场不再显示该 Skill（旧版本被隐藏）
- 用户体验差——更新提交后，旧版本"消失"了

## 解决方案：版本分支策略

**核心思路**：当更新已发布的 Skill 时，创建一条**新的 pending 记录**，保留原 approved 记录不变。

### 数据模型变更

**Skill 表**：增加 `parent_id` 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| parent_id | INT, FK→Skill.id, NULL | 父版本 ID，用于标识版本关系 |

**Review 表**（可选扩展）：增加 `version` 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| version | VARCHAR(50) | 审核时该版本的版本号 |

### 版本链结构

通过 `parent_id` 形成版本链：

```
Skill v1.0 (approved, id=1)
  └── Skill v1.1 (pending, parent_id=1, id=2)
  └── Skill v2.0 (approved, parent_id=1, id=3)  ← v1.1 被拒绝后，用户重新提交 v2.0
```

### 版本修改记录展示

通过版本链，可以轻松展示每个 Skill 的版本历史：

```sql
-- 获取某 Skill 的所有版本（按时间倒序）
SELECT * FROM skills
WHERE id = :id OR parent_id = :id OR parent_id IN (SELECT id FROM skills WHERE parent_id = :id)
ORDER BY created_at DESC;

-- 或通过递归 CTE
WITH RECURSIVE version_chain AS (
  SELECT * FROM skills WHERE id = :root_id
  UNION ALL
  SELECT s.* FROM skills s
  JOIN version_chain vc ON s.parent_id = vc.id
)
SELECT * FROM version_chain;
```

### 审核记录展示

审核记录通过 `skill_id` 关联 Skill，可展示：
- 每次审核的版本号（通过 Review.version 或查询 Skill 获得）
- 版本变更时间线

```
Skill: MySkill
├── v1.0 (2024-01-01 approved by admin)
├── v1.1 (2024-01-05 rejected by admin: 缺少安装说明)
│   └── v1.2 (2024-01-06 rejected by admin: 格式不对)
│       └── v2.0 (2024-01-07 approved by admin)
```

### 业务逻辑变更

#### 创建新版本时（`skill.service.ts` 的 `update` 方法）

```
IF skill.status === 'approved' THEN
  1. 复制 skill 创建新记录 newSkill
  2. newSkill.parent_id = skill.id
  3. newSkill.status = 'pending'
  4. 新记录包含用户提交的更新内容
  5. 原 skill 保持不变（status='approved'）
ELSE
  直接更新原记录
END
```

#### 审核通过时（`review.service.ts` 的 `approve` 方法）

```
IF skill.parent_id IS NOT NULL THEN
  1. 将父版本 skill.parent_id 的 status 改为 'superseded'
  2. 将当前 skill 的 parent_id 设为 null（成为正式版本）
END
skill.status = 'approved'
```

#### 审核拒绝时（`review.service.ts` 的 `reject` 方法）

```
skill.status = 'rejected'
skill.reject_reason = reason
-- 父版本不变
```

#### 查询逻辑

**广场列表**：
```
status = 'approved' AND (parent_id IS NULL OR parent_id NOT IN (SELECT id FROM skills WHERE status = 'approved'))
```
即：只显示最新的 approved 版本（不是被取代的版本）

**管理员待审核列表**：
```
status = 'pending'
```
（全部显示）

**用户的 Skill 列表**：
```
author_id = :userId
```
（显示用户所有版本，可通过 parent_id 展示版本关系）

#### 前端

- [ ] 个人中心：展示版本历史（可选，通过 parent_id 折叠/展开）

### 优点

1. **用户体验好**：更新期间旧版本保持可见
2. **版本历史清晰**：`parent_id` 可追溯完整版本链
3. **审核记录完整**：可展示每个版本的审核状态
4. **数据完整性**：不会丢失任何版本

### 缺点

1. 数据量增长更快（每次更新创建新记录）
2. 查询需注意排除被取代的版本
3. 需要处理数据库迁移

---

## 总结

此方案完全支持在审核记录中展示每个 Skill 的各版本修改记录。只需：

1. 在 Skill 列表中通过 `parent_id` 展示版本关系
2. Review 表可选择添加 `version` 字段记录审核时的版本号（方便直接查询）
